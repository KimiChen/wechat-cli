# wechat-cli 重构优先级清单

这份清单按 `P0` 到 `P3` 排序，原则是先修复会影响可用性和可信度的问题，再做结构化重构，最后补齐工程化和发布链路。

## P0 先止血：修复明显缺陷，补最小回归保护

- [ ] 修复 `members --format text` 的运行时错误。
  当前文本分支直接使用了未定义变量 `m`，这是一个会立刻暴露给用户的缺陷，应优先修复。

- [ ] 让多聊天 `search` 明确返回“聊天存在但没有消息表”的情况。
  现在 `resolve_chat_contexts()` 会收集 `missing_tables`，但命令层没有把这部分反馈给用户，容易造成“部分结果静默缺失”。

- [ ] 收紧“吞异常”逻辑，至少在 `stats` 和消息聚合路径里返回可见的失败信息。
  当前有多处 `except Exception: pass`，这会让统计结果和查询结果在出错时看起来像“成功但为空”，排查成本很高。

- [ ] 为核心命令补一组最小 smoke tests。
  优先覆盖 `init` 之外的只读命令入口、参数校验、空结果和错误输出，先建立“改动后不把 CLI 搞坏”的基础护栏。

## P1 整理结构：把共享状态和重复逻辑收拢

- [ ] 去掉 `core.contacts` 中的模块级全局缓存，改为绑定到 `AppContext` 或专门的缓存对象。
  现在 `_contact_names`、`_contact_full`、`_self_username` 是进程级全局状态，后续一旦支持多账号、长进程或测试并行，会比较容易串数据。

- [ ] 抽取 `session.db` 相关重复逻辑，统一 `sessions`、`unread`、`new-messages` 的查询和格式化流程。
  这三条命令有大量重复代码，后续修一个分支时很容易漏掉另外两个分支。

- [ ] 给 `contacts`、`messages`、`favorites` 建立更明确的“仓储层/查询层”边界。
  目标是让 `commands/` 只负责参数和输出，SQL 查询、数据转换、显示名解析分别沉到更稳定的模块中。

- [ ] 统一命令返回结构和失败信息模型。
  比如 `failures`、`count`、`scope`、`offset/limit` 这类字段可以做成一致约定，方便人用，也方便 Agent 调用。

## P2 提升可靠性与安全性：处理缓存、解析和性能问题

- [ ] 重构 `DBCache` 的缓存命名和生命周期管理。
  现在缓存文件名只基于相对路径，且长期留在系统临时目录；建议加入账号/数据目录维度，并提供更明确的清理策略。

- [ ] 评估是否需要为解密后的数据库增加过期、清理或显式 opt-in 持久化机制。
  当前实现对性能友好，但从隐私和多账号隔离角度看还不够明确。

- [ ] 优化消息表发现与搜索流程，减少重复扫描所有消息库的成本。
  目前多处逻辑会遍历全部 `message_*.db` 并重复构造上下文；可以考虑建立索引缓存或一次扫描复用结果。

- [ ] 梳理媒体路径解析策略，明确“精确定位”和“近似猜测”的边界。
  现在 `--media` 的体验已经有价值，但图片/语音/视频路径解析仍偏启发式，后续适合单独抽象和补测试。

- [ ] 重新定义 `new-messages` 的语义。
  当前更接近“会话级最近变更”，不是严格意义上的“逐条新消息流”；要么调整实现，要么把语义写清楚。

## P3 补工程化：让发布、验证和维护成本下降

- [ ] 对齐 Python 包、npm 主包和各平台包的版本号。
  当前主包是 `0.2.4`，部分平台包还是 `0.2.0`，发布链路容易出现认知和安装上的不一致。

- [ ] 修正 npm wrapper 中的错误提示与包名文案。
  例如安装失败提示里的包名应与真实发布名保持一致，减少用户排障摩擦。

- [ ] 增加基础 CI。
  最低限度建议包含：安装依赖、`compileall`、测试、打包 smoke check。

- [ ] 补开发者文档。
  建议写清楚项目分层、微信数据库目录约定、消息表映射规则、缓存目录、发布流程和已知兼容性边界。

## 建议执行顺序

1. 先完成 `P0`，把明显 bug 和“静默失败”问题收住。
2. 再做 `P1`，把共享状态和重复代码整理掉，为后续功能扩展减负。
3. 然后处理 `P2`，重点提升缓存安全性、解析稳定性和搜索性能。
4. 最后补 `P3`，让测试、发布和协作成本降下来。
